# churn-intervention-playbook (이탈 대응 플레이북 생성기)

AX 인재전쟁 2026 예선 — 채널코퍼레이션(채널톡) 트랙. 카카오페이증권 플러그인과 **완전 별도** 제출물.

---

## 1. 문제와 공개 근거

**(공개 사실)**
- 장바구니 이탈은 커머스의 가장 크고 보편적인 문제다 — 평균 이탈률 **70.22%** (Baymard, 50개 연구 평균, 2025-09-22, 무료 노출). 이탈 사유도 공개돼 있다(추가비용 39%·배송지연 21%·회원가입 강제 19%·결제복잡 18% 등, 단일 리스트).
- 채널톡은 이 문제를 **AI로 정면 겨냥**한다. 공개 방송(AX 인재전쟁 4화)에서 CSO는 "이탈 고객 → AI 전화 인터뷰 → 쿠폰"을 제품 방향으로 제시했다. (근거: `references/public-sources.md` V1)

**(우리 판단 — 공개 사실이 아니라 우리가 세운 통찰)**
- 그 서사를 그대로 자동화하면 **"이탈 = 쿠폰"이라는 반사적 대응**이 된다. 쿠폰은 마진을 잠식하고(원래 살 사람에게 낭비), 절약 지향 구매자를 선별 보상하며, 제품·UX·배송이 원인인 이탈은 쿠폰으로 못 고친다.
- 그래서 이 플러그인의 테제는 **"이탈 대응의 기본값은 쿠폰이 아니라 '왜 떠났는지 먼저 아는 것'"** 이다. 이 판단 규율이 제네릭 CRM 자동화와 이 도구를 가른다.

## 2. 무엇을, 누가, 어떤 상황에서

**채널톡 AX 컨설턴트**가 커머스 고객사의 **이탈(장바구니·결제·재구매) 상황을 진단할 때** 쓔다. 익명·공개안전 시나리오를 넣으면(사적 고객데이터 금지) 세그먼트별 개입을 판단하는 **플레이북**을 생성한다. 여러 고객사에 같은 판단 로직을 반복 적용하므로 **재현 가능한** 판단을 얻는다.

## 3. 어떻게 작동하나 — 원인-우선 3게이트

각 이탈 세그먼트를 순서대로 통과시킨다.

- **G1. 원인을 아는가?** 모르면 → 🎙️ **전화 인터뷰 먼저**(채널톡의 고유 방향으로 원인을 싸게 학습). 쿠폰 아님.
- **G2. 쿠폰으로 고쳐지나?** 제품·UX·배송 원인이면 → 🛠️ **제품팀 피드백**. 쿠폰은 증상만 덮음.
- **G3. 경제성이 되나?** 마진 잠식·절약 지향 세그먼트면 → ⛔ **개입 안 함**("좋은 이탈"도 존재).
- 세 게이트 **모두 통과** → 🎫 **선별 쿠폰** + holdout 병기 + **손익분기 임계치**(`breakeven.py`가 객단가·마진·할인폭으로 `T=d/(m−d)` 계산; d≥m이면 자동 ⛔).

이 "이탈≠쿠폰" 규율을 `check_playbook.py`가 **코드로 강제**한다 — 형식(3게이트·출처ID·holdout·손익분기)에 더해 실질검사: (a) 인용 출처가 `public-sources.md`에 **실존**(날조 방지), (b) 🎫인데 게이트 ❌면 **모순** FAIL·우선순위는 🎫만, (c) 가드레일 완비. 위반 시 exit 1.

## 4. 산출물 예시

실제 리포트 예시 2종(둘 다 린터 통과):
- **`examples/sample-playbook.md`** — 패션 D2C 기본 데모.
- **`examples/sample-playbook-counterintuitive.md`** — "판단"을 체감시키는 반직관 예시(고가치·원인불명 → 쿠폰 아닌 🎙️ 인터뷰 / 고이탈 bad-fit → ⛔ '좋은 이탈').

8섹션(원인진단 → **개입 라우팅 표(핵심)** → 개입 안 함이 정답인 세그먼트 → 우선순위(🎫만) → **🎬 영상 반사 대비** → 확인 필요 → 신선도 → 출처)이며 모든 판정에 출처 ID가 붙는다.

## 5. 설치·실행 (Codex 플러그인)

이 플러그인은 [Codex 플러그인 규격](https://developers.openai.com/codex/plugins)을 따른다(`src/.codex-plugin/plugin.json` + `skills/<이름>/SKILL.md`).

1. `src/`를 Codex가 플러그인을 탐색하는 위치에 둔다(공식 문서 참고).
2. 스킬 **`churn-intervention-playbook`** 을 호출한다. 익명 시나리오(업종/객단가대/관찰 이탈지점/세그먼트 힌트)를 주거나, 입력 없이 호출하면 붙박이 **합성 데모**로 구동된다.
3. `SKILL.md` 지침대로 3게이트를 적용해 8섹션 플레이북을 생성한다.

**생성 후 자기검증 (Python 3 표준 라이브러리만, 의존성 0):**
```bash
python src/skills/churn-intervention-playbook/scripts/check_playbook.py <리포트.md>
python src/skills/churn-intervention-playbook/scripts/breakeven.py <V> <m> <d>
python -m unittest discover -s src/skills/churn-intervention-playbook/scripts -p "test_*.py"
```
- 린트 통과 시 exit 0, 규율 위반 시 **exit 1**(FAIL 사유 출력).
- 한국어 Windows(cp949) 콘솔에서도 크래시하지 않도록 출력 인코딩을 UTF-8로 고정한다.

## 6. 어떻게 검증했나

- **코드**: 단위 테스트 **17개 통과**.
- **적대적 테스트**: 나쁜 리포트 8종으로 린터가 실제로 잡는지 실증.

## 8. 정직한 한계

- `check_playbook.py`는 구조·정합성 린트다 — 경제성 판단의 진위는 검증하지 않는다.
- G3 근거(P1·P2)는 초록 범위의 판단 원리로만 인용.
- 데모 시나리오는 합성/예시이며 사적 고객데이터가 아니다.
